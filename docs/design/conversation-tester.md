# 会話テスター + AI同士会話 サブシステム — 設計

- Status: Draft（設計確定前）
- 関連: [ADR-0003](../adr/0003-pipecat-cloud-pipeline.md)（Pipecat パイプライン基盤）
- Linear: AIC-7（予定）
- 最終更新: 2026-06-05

## Context（なぜ作るか）

2つの動機が1つの仕組みに収束する:

1. **近期: 会話品質の自動検証（人手レス）**。現状、音声エージェントの変更は実音声/ブラウザでしか
   検証できず、「起動は正常だが client が話すと LLM が死ぬ」型のバグ（prompt caching / 挨拶変更で
   実際に2回プロダクションを壊した）をデプロイ前に捕まえられない。**AI が話しかけ、AI judge が
   不自然さを採点**すれば、人手を介さずデプロイ前に回帰検出できる。
2. **将来: コンテンツ**。「**AI 同士が会話するのを聴衆が聴く**」体験。テスターの interlocutor が
   そのまま「もう一人のキャラ」になり、聴衆 join で成立する。

→ 使い捨てスクリプトでなく、**音声レベルで本物の会話を回す再利用可能なサブシステム**として組む。

## 確定した方針（ユーザー選択, 2026-06-05）

- **接続: 音声レベル × 2 エージェント**（STT/TTS/ターンテイキングまで本物。content にそのまま発展）
- **判定: AI judge 自動評価**（ログ+指標を LLM が採点し不自然箇所を具体指摘）
- **シナリオ: 目標駆動 improv**（役割+目標を与え LLM が即興。途中割り込み等も指示可能）

## アーキテクチャ

```
            ┌──────────────── Daily ルーム ────────────────┐
            │                                              │
  [あい: 本番エージェント] ⇄ 音声(WebRTC) ⇄ [interlocutor: 対話相手AI]
            │  (Pipecat Cloud)                  (役割/目標で improv)        │
            └───────┬──────────────────────────────┬───────┘
                    │ 転写+遅延指標                  │ 転写+遅延指標
                    ▼                                ▼
              [Session Director] ── 会話ログ統合(話者・時刻) ──▶ [AI Judge]
               ルーム作成/両者入室/先手/N ターン/記録            自然さ・人格一貫性・
                                                              ターンテイキング・遅延・
                                                              整合性を採点+不自然箇所指摘
```

### A. interlocutor（対話相手 AI）= 役割パラメータ化したエージェント
- **bot.py を再利用**（DRY）。`ROLE`/`PERSONA_PROMPT`/`ELEVENLABS_VOICE_ID`/`SCENARIO` を env/session
  で差し替え、**あいとは別の声・別人格**にする。あい本番コードと同じパイプライン（Daily→STT(ja)→
  LLM→TTS→Daily）。
- 目標駆動 improv: system に「あなたは <役割>。次の目標に沿って自然に会話する: <目標群>。
  読み上げ前提で簡潔に」。シナリオに barge-in 指示（例: 相手の発話途中で一度割り込む）も入れられる。

### B. Session Director
- Daily ルームを用意し、**あい + interlocutor を同じルームに入室**させる。
- 当面の実装: あいは Pipecat Cloud、**interlocutor は director がローカル実行**して同ルームに join
  （`DAILY_API_KEY` で参加）。将来は interlocutor も Cloud 化。
- 先手（kickoff）・最大ターン数・終了条件を制御。会話を**記録**（両者の転写＋発話/応答時刻）。

### C. AI Judge（評価）
- 統合した時系列トランスクリプト（話者・時刻・遅延）を LLM が**ルーブリック採点**:
  - 自然さ（日本語として不自然な言い回し/抑揚指示の破綻）
  - 人格一貫性（あいのペルソナ維持・OOC 無し）
  - ターンテイキング（早すぎる割り込み・不自然な沈黙・かぶり）
  - 遅延体感（応答までの間）
  - 整合性（話が噛み合っているか・自己紹介ループ等の劣化）
- 構造化出力（各軸スコア + 具体的な問題箇所の引用）。回帰検出に使う。
- 異種ベンダー（DeepSeek/Codex 等）を混ぜて groupthink を避ける余地あり（ADR-0001 の精神）。

## 二重の価値
- **テスト**: デプロイ後、director で AI 会話を回し「応答が返るか」+ judge スコアを**私が先に確認**してから
  人間検証に渡す → LLM を殺す系バグをデプロイ前後で機械的に捕捉。
- **コンテンツ**: interlocutor を「第2キャラ」に、judge を「進行役」に読み替えると、複数キャラ会話 +
  聴衆 join（listen-only）の体験になる。

## 段階的構築
- **Phase A**: bot.py を role パラメータ化（あい/interlocutor を同一コードで）。単体で interlocutor 起動確認。
- **Phase B**: Director（ローカル）— あいのルームに interlocutor を join、kickoff + N ターン + 転写記録。
- **Phase C**: AI Judge — トランスクリプト採点（ルーブリック・構造化出力）。CI 的に「回帰スコア」を出す。
- **Phase D（将来）**: 複数キャラ + 聴衆 join、進行役ディレクタ、録音/配信。

## リスク / 未確定
- **二者のターンテイキング衝突**: 両エージェントが VAD/割り込みを持つため、同時発話やデッドロックの恐れ。
  対策案: 明示 kickoff + 「相手の発話終了を待ってから話す」既定 + 役割で開始遅延に差。barge-in は専用
  シナリオでのみ誘発。
- **2 つの Cloud エージェントを1ルームに**: 当面 interlocutor はローカル実行で回避。Cloud 同室は別途検証。
- **トランスクリプト取得**: 各エージェントが転写を外部に push する仕組み or Daily の transcription を使う。
- **コスト**: 音声2系統 + judge で API コスト増。テストは短いシナリオで。
