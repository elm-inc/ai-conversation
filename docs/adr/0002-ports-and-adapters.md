# ADR-0002: 外部依存を ports & adapters で抽象化し差し替え可能にする

## ステータス

採択 (2026-06-03)

## 文脈

[ADR-0001](0001-cascaded-orchestration-architecture.md) の cascaded 構成では、STT・LLM・TTS・
ターン検出・記憶・埋め込み・音声トランスポートといった外部サービス/モデルに依存する。
これらは **2026 年時点で進化が速く、検証によって別物に差し替わる可能性が高い**
（例: STT のベンダー変更、日本語ターン検出モデルの自作版への置換、埋め込みモデルの更新）。

ベンダー固有の癖（partial transcript の不安定さ、遅延のばらつき、flush 挙動、日本語特性）が
オーケストレーションの中核ロジックに漏れ出すと、差し替えのたびにコアが壊れ、ロックインされる。

## 決定

**ports & adapters（ヘキサゴナル）** を採用し、
**ベンダー非依存の安定コア**と**交換可能なアダプタ**を分離する。

交換可能なポート（安定インターフェース）:

| Port | 責務 |
|---|---|
| `STTProvider` | 音声フレーム → partial/final transcript（時刻・確信度・**健全性フラグ**付き） |
| `TTSProvider` | テキストchunk → 音声フレーム（interrupt/flush、**声優音源の許諾範囲 capability・監査ログ**） |
| `LLMProvider` | messages+persona+cacheヒント → token stream（TTFT 露出） |
| `TurnDetector` | (partial, 音響特徴) → {complete/incomplete/backchannel/barge-in} 確率 |
| `MemoryStore` | 構造化関係状態の書込/検索（競合解決・忘却・PII マスキング） |
| `EmbeddingProvider` | text → vector |
| `AudioTransport` | WebRTC/WS の音声 I/O |

設計原則:
- **標準データ型を固定**（音声フレーム・transcript イベント・token イベント）。差し替えがコアに波及しない。
- **アンチコラプション**: ベンダー固有の癖はアダプタで吸収し、コアには正規化済みイベントのみ渡す。
- **capability 宣言**: 各 Provider が能力（partial 可否、barge-in flush 遅延 等）を宣言し、コアはフォールバック。
- **計測フックをポート境界に内蔵**: 同一 IF 背後で複数ベンダーをベンチ比較できる。
- **IF は「凍結」ではなく加法的バージョニング + optional capability** で進化させる（破壊的変更を避ける）。

## 理由

- ユーザー方針: 「個々の技術スタックや外部サービスは検証で変える可能性があるため、抽象化レイヤーをもうけた上で実装したい」。
- cascaded がテキスト段を露出する利点（[ADR-0001]）を活かし、各段をポート背後でベンチ→差し替えできる。
- 声優音源の法的リスク（人格権侵害）を `TTSProvider` アダプタのガードレール（許諾範囲・内容フィルタ・監査ログ）で構造的に防ぐ。
- STT の異常出力（ノイズ誤認識・繰り返し）を `STTProvider` の健全性ゲートで止め、投機応答の暴走を防ぐ。

## 検討した代替案

### 代替案 A: ベンダー SDK を直接コアから呼ぶ（抽象化なし）
- Pros: 初期実装が速い、ベンダー固有機能をフルに使える。
- Cons: 差し替えのたびにコア改修、ロックイン、テスト困難、癖がコアに漏れる。
- 不採用理由: 「検証で差し替える」という前提と真っ向から衝突する。

### 代替案 B: 単一フレームワーク（Pipecat/LiveKit）の抽象に全面依存
- Pros: 配管を自作せずに済む、エコシステムが使える。
- Cons: フレームワークの抽象に縛られ、日本語ターンテイキング等の自作コア要件と齟齬。
- 不採用理由: フレームワークは `AudioTransport` ポート背後の一アダプタとして使い、コアは独立に保つ。

## 帰結

### Pros
- 検証ベースでベンダーを選び差し替えられる。コアの安定性とテスト容易性。法的・健全性ガードレールをアダプタに集約。

### Cons
- 初期にポート粒度・標準イベント型を設計するコスト（Phase 0 の最優先タスク）。
- 過度な抽象化は YAGNI リスク。実際に差し替えうる軸に絞る。

### 実機検証 / 将来の検討事項
- Phase 0 でポート IF と内部標準イベント型を確定（凍結ではなく v1 として加法進化前提）。
- `AudioTransport` のフレームワーク選定（Pipecat vs LiveKit）は交換コストが高いため初期選定を慎重に。
