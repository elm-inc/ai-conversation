# ADR-0001: 中核アーキテクチャに cascaded + 自作オーケストレーション層を採用

## ステータス

採択 (2026-06-03)

## 文脈

生成音声によるキャラクター対話・コンパニオンサービスを構築する。技術的ブレイクスルーは
音声生成そのものではなく「人の発話 → AI判定 → 自然な応答」の対話制御にある。要件と制約:

- **声(TTS)は権利クリアな声優音源 × ElevenLabs に固定**したい（キャラの声が中核価値）。
- STT も既存サービスを利用する前提。
- ゼロベースで構築が必要なのは中間のオーケストレーション層（ターンテイキング・意図理解・人格・記憶）。
- 中核 KPI: ①発話終了→応答音声 体感 < 800ms ②発話意図を汲んだ正確な応答 ③一貫した人格・関係性。

音声対話のアーキテクチャは大別して 2 系統ある:
- **cascaded**: STT → LLM → TTS を各段ストリーミング接続。テキスト段が露出。
- **native speech-to-speech (S2S)**: 音声内蔵モデル（OpenAI Realtime 等）が一気通貫。最低遅延・自然な間。

## 決定

**cascaded（STT → LLM → TTS）+ 自作の対話オーケストレーション層**を中核アーキテクチャとする。

- 各段はストリーミングで接続し、オーバーラップ実行で合算ではなく最長経路の遅延に抑える。
- ターンテイキング（FSM: LISTEN/THINK/SPEAK/IDLE）・意図理解・先読み応答・人格・長期記憶は
  ベンダー非依存の自作コアとして実装する。
- 投機応答は「計算のみ」走らせ、**音声出力は semantic endpoint 確定でゲート**する（確定前の投機音声は出さない）。

## 理由

- **声を声優音源に固定する要件**が native S2S と根本的に衝突する。S2S は音声をモデル内部で生成するため、
  特定声優の権利クリア音源を TTS として差し込めない。cascaded なら TTS 段を ElevenLabs に固定できる。
- テキスト段が露出することで、人格制御（persona 注入）・長期記憶の検索注入・デバッグ・観測が可能になる。
- 各段を独立に差し替え・ベンチでき、特定ベンダーにロックインされない（[ADR-0002](0002-ports-and-adapters.md)）。
- 2026 年時点でも cascaded はプロダクション主流。ストリーミング最適化で sub-1s は達成可能。

## 検討した代替案

### 代替案 A: native speech-to-speech（OpenAI Realtime / Gemini Live 等）
- Pros: 最低遅延（~0.8s）、間・相槌・割り込みが自然、実装が単純。
- Cons: **独自TTS声に差し替え不可**（声優音源を使えない）、人格の細かい制御・記憶注入が弱い、ベンダーロックイン。
- 不採用理由: 「声優の声をキャラにする」中核要件を満たせない。これは妥協不可。

### 代替案 B: ハイブリッド（S2S で聞き取り/ターンテイキング、発話は既存TTS）
- Pros: ターンテイキングの自然さを S2S から得つつ発話は声優音源。
- Cons: 実装が複雑、2系統のモデルを協調させる難度、コスト増。
- 不採用理由: 初期は複雑性が見合わない。ターンテイキングは自作コアで十分狙える。将来再検討の余地は残す。

## 帰結

### Pros
- 声優音源を TTS に固定できる。人格・記憶・観測性を作り込める。各段差し替え自由。

### Cons
- 段間オーバーラップ・先読み・レイテンシ隠蔽を自作しないと遅延が積み上がる（800ms 達成に工夫が必要）。
- 日本語 semantic endpointing を自前で精度確保する必要がある（最大の技術リスク）。

### 実機検証 / 将来の検討事項
- Phase 0 で baseline 遅延を実測し 800ms 予算の妥当性を検証。
- ハイブリッド案は将来、ターンテイキング精度が頭打ちになった場合に再検討。
